En max-heap är ett **komplett binärt träd** som har regeln att **varje nods värde är ≥ alla barns värden**. Detta gäller för en **max-heap** (vilket är det vi normalt avser när vi skriver heap). För en **min-heap** gäller det omvända avseende regeln, dvs att föräldern har lägre värde ≤ än alla barns värden.

Kontrollera att träden uppfyller både kravet på att vara kompletta och regeln mellan barn och förälder. Såg du att det från vänster är en min-heap och sen två max-heapar?

Genom att använda heap:ar kan vi alltid vara säkra på att vi har det högsta (mest prioriterade) elementet i roten, vilket vi snabbt kan komma åt. Detta är relevant när akuta/prioriterade fall/ärenden måste identifieras. Det är även relevant vid vissa algoritmer i samband med grafer som vi kommer behandla senare.

ADT:n Heap har operationerna: 
- sätta in ett element
- ta bort elementet med högst värde/prioritet
- "kika" på elementet med högst värde/prioritet

## Insättning

Vi sätter in ett nytt värde som nästa löv i det kompletta trädet. Nu kan detta nya värde göra så att regeln att föräldern har ett högre värde än barnen inte är uppfyllt. Alla övriga noder i trädet uppfyller detta så det är bara värdet i den nya noden som kan göra att regeln bryts. 
Detta måste åtgärdas i så fall. Vi kontrollerar den nya nodens värde mot dessa förälder. Om regeln inte är uppfylld byter vi värdet. Vi vet med säkerhet att efter bytet så har föräldern ett större värde än båda barnen (eftersom det nya värdet var större än den tidigare föräldern).  
Nu behöver vi kontrollera mot den nya föräldern (en nivå högre) och vi byter återigen värdet om regeln inte är uppfylld. Detta fortgår tills dessa att regeln uppfylls. Detta kan innebära att vi når roten vilket inträffar om det nya värdet är det största av alla.

Första värdet blir automatisk roten.

## Borttagning av högsta värdet

Högsta värdet finns alltid i roten. Tar vi bort detta behöver vi återställa strukturen så att trädet blir komplett och dessutom följer regeln för heapen. 

När rotens värde är borttaget ersätts det med det löv som finns längst till höger på nedersta nivån. På så sätt bibehålls strukturen av det kompletta trädet.
Nu återstår att se till att regeln uppfylls. Vi vet att det enda värde som gör att regeln bryts är det nya rotvärdet. Nu börjar vi uppifrån och kontrollerar det nya värdet mot de båda barnen. Är det nya värdet mindre än det största av barnens värden behöver vi byta med detta (då blir den nya nodens värde större än båda barnen vilket måste gälla).  
När det är gjort fortsätter vi på samma sätt med de nya barnen och byter med det största av dessa om regeln inte är uppfylld.

Detta är trivialt då det största värdet är i roten och detta är hittas direkt och levereras.

## Representation

En heap lämpar sig väl att lagra i en array. Anledningen är att det binära trädet är komplett. Om vi placerar roten index 0, dess vänstra barn på index 1, dess högra barn på index 2 osv så att vi fyller på arrayen nivå för nivå i trädet så blir det inga "luckor" i arrayen.

<br></br>
<Figure
    src={heaparray}
    caption="Lagring i array"

/>


I bilden ser vi "samma" kompletta träd där ett är svart och ett är rött. Det svarta innehåller heapens värden i noderna och det röda motsvarande index i arrayen. Under de båda träden ser vi hur heapen lagras i arrayen.

Det finns ett mönster avseende var (vilket index) en nods förälder och en nods barn är placerade. 

Om noden finns på index `k` beräknas index för 
- **föräldern** genom `(k-1)/2`
- **vänster barn** genom `2*k + 1`
- **höger barn** genom` 2*k + 2`

Observera att vid (k-1)/2 är det heltalsdivision som gäller.

Vi utgår t.ex från noden med värdet 20 som är lagrat på index 3.  
- Vi hittar dess förälder på index (3-1)/2 = 1. 
- Vänster barn finns på index 3*2 + 1 = 7
- Höger barn finns på index 3*2 + 2 = 8

### Insättning

När man sätter in ett nytt element, placeras det som löv. 
Detta ska sedan vandra uppåt i arrayen och byta med föräldern om deras inbördes ordning inte uppfyller regeln (föräldern ≥ barn). Denna del är lämplig att hantera i en hjälpmetod, t.ex kallad `perculateUp()`.

Vad vet vi och vad behöver vi utföra för att lösa detta?

Vi vet att 
- `endOfHeap` har index för platsen där det nya elementet ska placeras och detta ska därefter räknas upp med 1
- vi behöver en variabel `current` för att kunna vandra uppåt i heapen vilken sätts till index för det nya elementet
- vi kan beräkna index för föräldern
- vi behöver jämföra elementet på index `current` med förälderns element för att avgöra om regeln är uppfylld eller ej
- om regeln inte är uppfylld kan vi byta elementen och ändra `current` så att den får det index som beräknades för föräldern 
- vi fortsätter tills det nya elementet hamnat i roten eller om relationen med föräldern följer regeln  

**Testa dig själv**  

1. Skriv psedokod för `perculateUp()`  

    <details>
    <summary>Klicka här för att visa svaret</summary>   

    ```
    current = endOfHeap - 1
    parent = (current - 1)/2
    while current > 0 and heapItems[current] > heapItems[parent]
        swap(heapItems[current], heapItems[parent])
        current = parent
        parent = (current - 1)/2
    ```
    </details>
### Borttagning

Det är alltid rot-elementet som tas bort. Detta ersätts med lövet längst till höger på nedersta nivån. Lövet ska vandra nedåt i arrayen och byta med det största av de båda barnen om regeln (föräldern ≥ barn) inte är uppfylld. Denna del är också lämplig att hantera i en hjälpmetod, t.ex kallad `perculateDown()`.

Vad vet vi och vad behöver vi utföra för att lösa detta?

Vi vet att 
- `endOfHeap` har index för platsen efter det som lövet hämtas ifrån och ska minskas med 1
- vi behöver en variabel `current` för att kunna vandra nedåt i heapen, vilken sätts till 0 (roten-index)
- vi kan beräkna index för vänster och höger barn
- vi behöver bestämma det barn som har högst värde (obs kan vara endast ett barn när current är på näst nedersta nivån)
- vi behöver jämföra elementet på index `current` med det största barnet
- om regeln inte är uppfylld kan vi byta elementen och ändra `current` så att den får det index som beräknades för det största barnet
- vi fortsätter tills det nya elementet hamnat som löv eller uppfyller regeln

    ```
    current = 0
    leftChild = 2*current + 1
    biggest = leftChild
    while not done and leftChild < endOfHeap // endOfHeap har räknats ner innan
        if heapItems[2*current + 2] > heapItems[leftChild]
            biggest = 2*current + 2
        if heapItems[biggest] > heapItems[parent]
            swap(heapItems[biggest], heapItems[parent])
            current = biggest
        else
            done = true
    ```

## Sammanfattningsvis

En Heap är ett binärt träd som lämpar sig att använda när man snabbt behöver få tag i det element som har högst värde/prioritet.   
En (max)-Heap uppfyller två regler:  
- Det binära trädet är komplett.
- För varje nod gäller att förälderns värde är större än eller lika med barnens värden.  

För en min-Heap gäller det omvända mellan förälder och barn, att för varje nod är förälderns värde mindre än eller lika med barnens värden. 

En heap använder en array/sekvens som intern datastruktur vilket ger snabb access till förälder och barn till en nod eftersom index för dessa enkelt beräknas:
    - förälder finns på (index - 1)/2
    - vänster barn finns på 2*index + 1
    - höger barn finns på 2*index + 2
