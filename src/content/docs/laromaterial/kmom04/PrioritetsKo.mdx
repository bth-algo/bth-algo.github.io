I en prioritetskö (priority queue) spelar insättningsordningen ingen roll, men det som är garanterat är att det objekt osm har högst prioritet av de som finns i kön är det element som kommer att lämna kön vid nästa borttagning. En liknelse från verkligheten som ligger närmast är en sk VIP-kö. Den som är *viktigast*/*har högst status* är den som lämnar kön först oavsett när personen i fråga kom till kön.

Prioritetsköns beteende, dvs uppsättning operationer, motsvaras av

- enqueue - sätt in ett element i kön
- dequeue - ta bort det element som har högst prioritet
- peek - "kika" på det noden som har högst prioritet utan att ta bort det
- size - avgör storleken, dvs antalet element, i kön

Det finns dessutom ett krav på elementen som ska placeras i prioritetskön och det är att det ska vara möjligt att jämföra elementen. För att uppnå kravet på elementen använder vi det generiska interfacet (som vi har tittat på tidigare)

```csharp
public interface IComparable<in T>
{
    int CompareTo(T? other);
}
```

Metoden `int CompareTo(T? other)` returnera
- *< 0* om anropande objekt (this) är *mindre än* other
- *0* om anropande objekt (this) är *lika med/ekvivalent* other
- *> 0* om anropande objekt (this) är *större än* other

## Interface för Prioritetskö

Vi skapar ett generiskt interface med krav på att den generiska datatypen ska vara jämförbar.

```csharp
interface IPriorityQueue<T> where T : IComparable<T>
{
    void Enqueue(T elem);
    T Dequeue();
    T Peek();
    int Size();
}
```
Nu har vi sett till att det finns krav på metoden `CompareTo` för den generiska datatypen som ska hanteras i prioritetskön.

### Använda ADT Prioritetskö

Anta att interfacet `IPriorityQueue` har implementerats av den generiska klassen `PriorityQueue`. 

Vad förväntar vi oss av klassen PriorityQueue? 

- Att vi kan skapa en tom prioritetskö
- Att vi kan placera element i prioritetskön.
- Att vi kan ta bort element noden som har högst prioritet.
- Att vi kan kika på det element som har högst prioritet utan att ta bort det    

Vi exemplifiera med int som vi vet uppfyller kravet på den generiska datatypen, nämligen att `CompareTo` finns.
    
```csharp
IPriorityQueue<int> pq = new PriorityQueue<int>();

pq.Enqueue(222); // 222 är placerad i prioritetskön
pq.Enqueue(111); // 111 är placerad i prioritetskön
pq.Enqueue(444); // 444 är är placerad i prioritetskön
pq.Enqueue(333); // 333 är är placerad i prioritetskön

int res = pq.Dequeue(); // 444 har högst prioritet och tas bort från prioritetskön ...

res = pq.Peek(); // 333 har högst prioritet och returneras till res men tas inte bort
res = pq.Dequeue(); // 333 har högst prioritet och tas bort från prioritetskön ...
res = pq.Dequeue(); // 222  har högst prioritet och tas bort från prioritetskön ...

```

Vi exemplifiera även med Car (enligt ovan) som också uppfyller kravet på `CompareTo` 
    
```csharp
IPriorityQueue<Car> pqCars = new PriorityQueue<Car>();

pqCars.Enqueue(new Car("Lexus", 2019)); 
pqCars.Enqueue(new Car("BMW", 2015)); 
pqCars.Enqueue(new Car("Volvo", 2023)); 
pqCars.Enqueue(new Car("Nissan", 2007)); 

Car res = pqCars.Dequeue(); // bilobjektet motsvarande Volvon av årsmodell 2023 tas bort

res = pqCars.Peek(); // bilobjektet motsvarande Lexus av årsmodell 2019 returneras men tas inte bort
res = pqCars.Dequeue(); // bilobjektet motsvarande Lexus av årsmodell 2019 tas bort
res = pqCars.Dequeue(); // bilobjektet motsvarande BMW av årsmodell 2015 tas bort

```

### Överkurs: Representera prioritetskö

Nu är det dags att implementera ADT:n Prioritetskö. Vi behöver göra lite mer än när vi implementerade Stack coh Kö eftersom vi behöver kräva att elementen som placeras i prioritetskön är jämförbara.

```csharp
public class PriorityQueue<T>: IPriorityQueue<T> where T: IComparable<T>
{
    // ...
}

```

Precis som för ADT:erna Stack och Kö behövs det konstruktor och definition av metoderna som ingår i interfacet. Dessutom behöver vi hitta något sätt för en prioritetskö att hantera de element som placeras i den. 

### Intern datastruktur

Vi ska återigen använda alternativen **array** respektive **länkning** men vi behöver tänka till så att vi använder dessa på ett effektivt sätt.

#### Array 

Här är det lämpligt att bara lägga in ett nytt element efter de redan befintliga. Det ger bra effektivitet om vi samtidigt håller reda på detta index. Vi kan inte få samma effektivitet när vi ska ta bort det element som har högst prioritet eftersom vi inte vet var det finns. Alla element måste kontrolleras för att kunna avgöra vilket som har högst prioritet/värde.

#### Länkning 

I detta fall väljer vi att låta en enkellänkad listan vara sorterad baserat på elementens prioritet. Det innebär att vi vid varje insättning placerar in det nya noden på rätt plats enligt sorteringsordningen.

På samma sätt som för stack och kö använder vi en inre privat klass för att representerar noder, men denna gång behöver vi även se till att den generiska datatypen (här benämnd `E`) är jämförbar eftersom vi behöver orda noderna baserat på det som lagras i dem (som är `Data` av typen `E`). Vi använder därför `IComparable` även här. 

```csharp
private class Node<E> where E: IComparable<E>
{
    public E Data;
    public Node<E>? Next;
    public Node(E data, Node<E>? next = null)
    {
        this.Data = data;
        this.Next = next;
    }
}
```

#### Analys av tidskomplexiteten

Vi analyserar operationerna **Enqueue**. **Dequeue**, **Peek** och **Size** för de båda alternativ av intern datastruktur som vi gått igenom (array och länkning).

##### Array 

Analysen omfattar inte expandering av arrayen.  

- **Enqueue**: Arbetet innebär att placera ett element på ett känt/givet index och att räkna upp detta index (öka innehållet i variabeln med 1). Samma arbete utförs oavsett antalet element i arrayen, vilket ger **O(1)**.
- **Dequeue**: Arbetet innebär att 
    1. bestämma det index där elementet med högst prioritet finns
    2. ersätta detta med det som är sist (vilket är ett känt index)
    3. minska räknare med 1 
    4. returnera elementet

    Vad blir då kostnaden?  
    1. alla element måste kontrolleras => **O(n)**
    2. alltid samma oavsett antal element => **O(1)**
    3. alltid samma oavsett antal element => **O(1)**
    4. alltid samma oavsett antal element => **O(1)**

    Vi vet att det är den dominerande termen som avgör, varför vi får **O(n)**.

    **Testa dig själv**
 
    Vilken tidskomplexitet i Ordo-notation blir det för `Peek` och för `Size`?

    <details>
    <summary>Klicka här för att visa svaret</summary>
    O(n) för `Peek` med motsvarande resonemang som för `Dequeue`.  
    O(1) för `Size` eftersom det endast är en retur av innehållet i en variabel.
    </details>

**Anmärkning**: Genom att använda en medlemsvariabel som håller index för var det element med högst prioritet finns kan man få O(1) för `Peek`. Övriga operationer påverkas inte av detta tillägg. Fundera över vad det innebär i kod för metoderna.  

##### Länkning

- **Enqueue**: Arbetet innebär att 
    1. hitta rätt plats bland de redan sorterade genom att börja från början av listan och *vandra* framåt med en pekare
    2. infoga ett nytt element vid den pekare som *vandrat*
    3. öka antalet med 1

    Vad blir då kostnaden?  

    <details>
    <summary>Klicka här för att visa svaret</summary>  

    1. alla element måste kontrolleras i värsta fallet (och hälften i genomsnitt)=> **O(n)** både värsta fall men även i medelfallet
    2. alltid samma oavsett antal element => **O(1)**
    3. alltid samma oavsett antal element => **O(1)**

    Vi vet att det är den dominerande termen som avgör, varför vi får **O(n)** i både värsta fall och i medelfallet.
    </details>


- **Dequeue**: Arbetet innebär att hämta elementet i första noden, plocka bort noden, räkna ner antalet samt returnera elementet. Det blir alltså **O(1)**

- **Peek**: Arbetet innebär att returnera elementet i första noden, vilket ger **O(1)**.  

- **Size**: Arbetet innebär att returnera innehållet i räknaren som håller antalet element, **O(1)**.

## Sammanfattning

- ADT Prioritetskö garanterar att det element som har högst prioritet kommer att vara det element som lämnar kön.
- Beteendet, dvs de operationer som ADT:n Prioritetskö har är
    - enqueue - sätt in ett element i kön
    - dequeue - ta bort det element i kön som har högst prioritet
    - peek - "kika" på det element i kön som har högst prioritet utan att ta bort det
    - size - leverera hur många element prioritetskön innehåller
- För att implementera en prioritetskö kan man använda
    - array - där elementet inte ordnas utan arrayen är oordnad/osorterad
    - enkellänkning - där elementen i listan är ordande efter prioritet.
- Vid användande av array blir kostnaden linjär **O(n)** vid borttagning (och peek) och i övrigt konstant.
- Vid användande av länkning blir kostnaden linjär **O(n)** vid insättning och i övrigt konstant.
